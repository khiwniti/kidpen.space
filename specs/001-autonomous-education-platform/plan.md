# Implementation Plan: Kidpen Autonomous Education Platform

**Branch**: `001-autonomous-education-platform` | **Date**: 2026-04-25 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-autonomous-education-platform/spec.md`

**Note**: This plan was generated through the Spec Kit workflow after initializing Specify and establishing the Kidpen constitution.

## Summary

Merge the production autonomous-agent SaaS runtime in `/Users/khiwn/kidpen/kidpen.space` with the Thai-first education product prototype in `/Users/khiwn/kidpen-space`. The production repo remains the canonical runtime. The prototype is treated as product/UX/prompt/design source material. Implementation proceeds through safe vertical slices, starting with the student Thai Socratic tutor loop: auth/consent -> dashboard -> tutor chat -> production agent run/stream -> interaction log -> mastery update -> progress display.

## Technical Context

**Language/Version**: Python 3.11 backend; TypeScript/React frontend; SQL/Postgres migrations; Expo/React Native later for mobile reuse  
**Primary Dependencies**: FastAPI, Supabase/Postgres, Redis/Upstash, AgentPress, production agent pipeline, Daytona sandbox, Next.js/React, Tailwind/Radix, Stripe/RevenueCat, Langfuse/Braintrust-style evals  
**Storage**: Supabase/Postgres for production data; Redis for agent streaming/run state; sandbox storage for controlled computer-use artifacts; Google Drive appdata only where explicitly required for student-owned storage  
**Testing**: Backend pytest/e2e tests, frontend TypeScript/lint/build tests, browser/manual QA, AI eval datasets for Thai Socratic behavior and safety, RLS/authorization tests  
**Target Platform**: Web SaaS first; mobile-compatible API and UI paths; future Expo mobile reuse; sandbox workloads through Daytona  
**Project Type**: Multi-app SaaS monorepo with backend API, frontend web app, mobile app, shared packages, infra, and autonomous agent runtime  
**Performance Goals**: Core tutor loop usable in under 3 minutes manually; streamed first visible response target under 5 seconds when provider is healthy; dashboard initial load target under 3 seconds on common broadband; sandbox lab cold start tracked separately and hidden behind progress UI  
**Constraints**: Thai-first student UX; PDPA/child safety by design; no direct homework answer leakage; no unrestricted student computer-use; use production runtime instead of prototype direct LLM/SQLite/demo auth; preserve existing local changes through audit before large migrations  
**Scale/Scope**: MVP covers one complete student tutor loop plus data/model foundations for teacher/parent/lesson/homework/lab expansion; not all prototype widgets are migrated in first slice

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Existing production repo has many local changes | High | Source audit before migration; no bulk copy until diff reviewed |
| Duplicate nested checkouts and possible embedded Git credential | High | Do not migrate `.git`; rotate token if real; document canonical repo |
| Prototype monolith tempts direct copy | High | Constitution prohibits direct monolith import; extract vertical slices only |
| Homework engine gives direct answers | High | Policy classifier, evals, tests, and prompt constraints before release |
| Student computer-use unsafe | High | Disable until tool policy/audit/RLS tests pass |
| Parent/teacher visibility leaks private student data | High | RLS/relationship/consent tests and safe summary contracts |
| AI provider/sandbox downtime breaks learning loop | Medium | Graceful fallback and record preservation |
| Scope creep from 86 prototype widgets | Medium | MVP limited to core tutor loop and essential dashboard widgets |
