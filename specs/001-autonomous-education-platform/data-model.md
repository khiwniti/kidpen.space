# Data Model: Kidpen Autonomous Education Platform

## LearnerProfile

Represents a student using the platform.

Fields:

- `id`: stable learner profile identifier
- `user_id`: linked production auth user
- `tenant_id`: school/class/family tenant scope
- `display_name`: student-visible name
- `grade_level`: Thai grade or equivalent level
- `preferred_language`: defaults to Thai
- `guardian_consent_state`: active, missing, withdrawn, expired
- `accessibility_preferences`: reduced motion, font size, audio preferences
- `created_at`, `updated_at`

Relationships:

- Has many MasteryState records
- Has many LearningInteraction records
- Has many TutorThread records
- May have many GuardianRelationship records
- Belongs to tenant/class scopes

Validation:

- Student data access requires self, authorized teacher, authorized guardian, or admin scope.
- Student-facing features require active required consent where applicable.

## EducatorProfile

Represents a teacher or school educator.

Fields:

- `id`
- `user_id`
- `tenant_id`
- `display_name`
- `subjects`
- `class_ids`
- `permissions`
- `created_at`, `updated_at`

Relationships:

- Can create Assignments and Lessons
- Can view LearnerProfiles only within authorized tenant/class scope

Validation:

- Teacher visibility is bounded by tenant/class assignment and consent/role policy.

## GuardianRelationship

Represents parent/guardian access to a learner.

Fields:

- `id`
- `guardian_user_id`
- `learner_id`
- `relationship_type`
- `verification_status`
- `visibility_scope`
- `consent_scope`
- `created_at`, `updated_at`, `revoked_at`

Relationships:

- Links parent/guardian user to LearnerProfile
- References ConsentRecord history

Validation:

- Parent access requires verified active relationship.
- Raw conversation visibility is disallowed unless explicitly consented and policy allows.

## Subject

Represents an education subject area.

Fields:

- `id`
- `key`: math, physics, chemistry, biology, cs, etc.
- `name_th`
- `name_en`
- `icon`
- `color`
- `description`
- `grade_range`
- `order`

Relationships:

- Has many KnowledgeComponent records
- Has many Lessons and Assignments

Validation:

- Student-facing display defaults to Thai name.

## KnowledgeComponent

Represents a tracked concept or skill.

Fields:

- `id`
- `subject_id`
- `key`
- `name_th`
- `name_en`
- `grade_level`
- `ipst_standard`
- `prerequisite_ids`
- `assessment_metadata`

Relationships:

- Has many MasteryState records
- Referenced by LearningInteraction, Lesson, Assignment, TutorThread

Validation:

- Knowledge components must belong to a subject and grade band.

## TutorThread

Represents a tutoring conversation.

Fields:

- `id`
- `learner_id`
- `subject_id`
- `knowledge_component_id`
- `agent_id`
- `latest_run_id`
- `title`
- `visibility_policy`
- `safety_flags`
- `created_at`, `updated_at`

Relationships:

- Has many messages through production thread/message system
- Has many LearningInteraction records
- Has many agent runs

Validation:

- Thread access follows learner/teacher/guardian/admin visibility policy.

## LearningInteraction

Represents evidence generated by chat, lesson, homework, quiz, or lab activity.

Fields:

- `id`
- `learner_id`
- `thread_id`
- `lesson_id`
- `assignment_id`
- `knowledge_component_id`
- `subject_id`
- `interaction_type`: chat, lesson_checkpoint, homework_check, quiz, lab
- `response_type`: question, hint, attempt, answer_check, explanation, completion
- `is_correct`
- `scaffolding_level`
- `response_time_ms`
- `evidence_summary`
- `safety_flag`
- `created_at`

Relationships:

- Updates MasteryState
- References TutorThread/Lesson/Assignment as applicable

Validation:

- Must be tied to a learner.
- Correctness may be null if not assessable.

## MasteryState

Represents a learner's current estimate for a knowledge component.

Fields:

- `id`
- `learner_id`
- `knowledge_component_id`
- `p_mastery`
- `confidence`
- `opportunity_count`
- `correct_count`
- `last_practiced_at`
- `last_evidence_interaction_id`
- `updated_at`

Relationships:

- Belongs to LearnerProfile and KnowledgeComponent

Validation:

- Unique per learner and knowledge component.
- `p_mastery` is between 0 and 1.

## Lesson

Represents structured interactive learning content.

Fields:

- `id`
- `subject_id`
- `knowledge_component_ids`
- `title_th`
- `title_en`
- `objective`
- `content_blocks`
- `checkpoint_items`
- `practice_items`
- `created_by_user_id`
- `source_type`: teacher_created, ai_generated, system
- `review_status`
- `created_at`, `updated_at`

Relationships:

- Can be assigned via Assignment
- Generates LearningInteraction records

Validation:

- AI-generated lessons require review status before broad classroom publication unless tenant policy allows draft use.

## Assignment

Represents teacher/system assigned work.

Fields:

- `id`
- `tenant_id`
- `class_id`
- `teacher_id`
- `subject_id`
- `knowledge_component_ids`
- `title`
- `instructions`
- `due_at`
- `allowed_tool_policy_id`
- `homework_solution_policy`
- `rubric`
- `status`
- `created_at`, `updated_at`

Relationships:

- Has many student submissions/interactions
- May launch ComputerLabSession

Validation:

- Tool policy is required if assignment permits lab/computer-use.

## ComputerLabSession

Represents a sandboxed agent/tool session.

Fields:

- `id`
- `learner_id`
- `assignment_id`
- `tenant_id`
- `sandbox_id`
- `agent_run_id`
- `tool_policy_id`
- `status`: pending, running, blocked, completed, failed, cancelled
- `artifact_refs`
- `started_at`, `ended_at`

Relationships:

- Has many AuditEvent records
- May produce artifacts/files

Validation:

- Student sessions require assignment context and active tool policy.

## ConsentRecord

Represents consent and privacy decisions.

Fields:

- `id`
- `learner_id`
- `actor_user_id`
- `actor_role`
- `scope`
- `status`: granted, withdrawn, expired
- `policy_version`
- `effective_at`
- `expires_at`
- `withdrawn_at`
- `created_at`

Relationships:

- Affects GuardianRelationship and data-use behavior

Validation:

- Consent changes must be auditable and immediately enforceable.

## AuditEvent

Represents privacy, safety, auth, billing, or tool-use event.

Fields:

- `id`
- `actor_user_id`
- `actor_role`
- `target_type`
- `target_id`
- `action`
- `result`: allowed, denied, failed
- `reason_code`
- `trace_id`
- `metadata`
- `created_at`

Relationships:

- References thread/run/session/assignment where applicable

Validation:

- Required for tool policy decisions, sensitive data access, consent changes, safety escalations, and billing-relevant agent runs.

## State Transitions

### TutorThread

- `created` -> `active` when first message is sent
- `active` -> `safety_review` when safety concern detected
- `active` -> `archived` by user/system retention policy

### Assignment

- `draft` -> `published` after teacher review
- `published` -> `closed` after due date or teacher action
- `closed` -> `archived` by retention policy

### ComputerLabSession

- `pending` -> `running` after sandbox allocation
- `running` -> `completed` after successful finish
- `running` -> `blocked` after policy denial that stops the run
- `running` -> `failed` after sandbox/tool failure
- `running` -> `cancelled` after user/system cancellation

### ConsentRecord

- `granted` -> `withdrawn` by guardian/student policy actor
- `granted` -> `expired` by expiration
- new `granted` record supersedes prior scope decisions according to policy version
